## 作業手順

### 1. 変更履歴ページを新設（Desktop 非依存・確実に作れる）
- 配置: `vault/06_リファレンス/` に新規 Markdown。slug: `reference/changelog`。
- 内容: v0.16.0 → v0.17.0 の利用者向け変更点と**破壊的変更の移行注記の正本**（`write_mode`→`write_approval`、`send_message` 削除など、設定画面に出ないがガイド読者が知るべき事項）。出典 URL（リリースノート/compare）を付す。
- frontmatter: `title` / `description` / `slug: reference/changelog` / `hermes_version: 0.17.0` / `status: needs-review` / `tags` / `sidebar.order`（リファレンス内の適切な位置）。
- `06_リファレンス/` は既存セクションなので **`astro.config.mjs` 変更不要**（autogenerate）。

### 2. 新機能の Desktop 到達性を確認（人間協働・Stop And Ask）
Wave 2/3 申し送りの各機能について、人間に「Desktop から到達できるか」を確認:
- 到達できる → 独立ページ化（手順 3）。
- CLI/別サーフェス限定 → 変更履歴ページに注記 + 公式リンク集に 1 行（独立ページ化しない）。

### 3. 新規ページ作成（`hermes-docs-writer`。到達できたものだけ）
候補（到達性が取れたものを採用。slug は ASCII・不変前提）:

| 候補ページ | slug | vault 配置 | 新セクション? |
|---|---|---|---|
| 変更履歴（手順1で作成済み） | `reference/changelog` | `06_リファレンス/` | 否（既存） |
| キーボードショートカット（リバインド可能） | `basics/keyboard-shortcuts` | `02_基本操作/` | 否（既存） |
| OS 通知（per-type）※独立画面なら | `settings/notifications` | `03_設定/` | 否（既存。実 UI 順に `sidebar.order` 挿入。既存 settings の order 再採番が要るなら docs-writer が frontmatter で調整） |
| ダッシュボード概要 + Secure Login | `dashboard/overview` | `07_ダッシュボード/`（新規） | **要**（新セクション） |
| プロファイルビルダー + マルチプロファイル | `dashboard/profiles` | `07_ダッシュボード/` | 要 |
| Skills Hub 刷新 | `dashboard/skills-hub` | `07_ダッシュボード/` | 要 |
| 連携の全体像 + セキュリティ前提 | `channels/overview` | `08_連携/`（新規） | **要** |
| iMessage（Photon `hermes photon login`） | `channels/imessage` | `08_連携/` | 要 |
| メッセージング各種（WhatsApp/Telegram 10.1/SimpleX/Raft） | `channels/messaging` | `08_連携/` | 要 |

各新規ページの必須事項:
- 本文に **H1 を書かない**（Starlight PageTitle が生成）。
- 公開基準: 前提・操作・期待結果・失敗時の確認先を含める。「準備中」ページを作らない。
- 新規は `status: needs-review`（実機確認後の昇格は別途）。
- WikiLink は**パス付き**（`[[06_リファレンス/03_公式リンク集|公式リンク集]]`）。
- slug は ASCII 小文字・数字・ハイフン・`/`。basename 重複を作らない。

### 4. 新セクション追加時のみ `astro.config.mjs` を更新（`starlight-config-deployer` に委譲）
- `dashboard/` `channels/` を採用した場合、`sidebar` 配列に対応セクションを追加:
  ```js
  { label: "ダッシュボード", items: [{ autogenerate: { directory: "dashboard" } }] },
  { label: "連携・チャネル", items: [{ autogenerate: { directory: "channels" } }] },
  ```
  挿入位置はサイト情報設計に合わせる（設定の後、リファレンスの前あたり）。
- slug を変えないので **`redirects`/`legacyRedirects` の変更は不要**。
- 既存セクション内のページ（changelog/keyboard-shortcuts/notifications）は autogenerate で拾われるので config 変更不要。

### 5. 検証
- `npm run build`（新セクションのルート生成・autogenerate 反映）。
- `npm run check:links`（新規ページの内部リンク・アンカー）。

## 完了条件（DoD・機械判定）

```bash
npm run preprocess     # => ContentValidationError なし（新規 slug/WikiLink 解決）
npm run build          # => dist 生成成功（新セクションのルート含む）
npm run check:links    # => 内部リンク・アンカー切れなし
# 変更履歴ページが存在し slug を持つ
grep -rl "slug: reference/changelog" vault/06_リファレンス/   # => 1件
```
加えて（人手確認）: 作成した新規ページは Desktop 到達性が取れたものだけ。到達不可の機能は変更履歴/公式リンク集の注記に留めた。新セクション追加時は sidebar に表示される。

## 失敗時の撤退

- build/check:links が落ちたら、新規ページの frontmatter（slug 必須・basename 重複）と WikiLink を確認。`astro.config.mjs` の sidebar 追加で `directory` 名が slug 先頭と一致しているか確認。
- 巻き戻し: 新規ファイルを削除（`git clean` ではなく個別 `rm`）、`astro.config.mjs` は `git restore astro.config.mjs`。
- 到達性が判断できない機能は作らずに次へ（安全側）。実行ログに「未作成・理由」を記録。
