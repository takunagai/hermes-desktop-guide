## 作業手順（実機協働・画面ごと）

`vault/06_リファレンス/02_バージョンと調査根拠.md` の手順に準拠し、**設定サイドバーを上から順に**進める。対象は 14 画面 + プロバイダーの 2 サブメニュー + ツールとキーの 2 サブメニュー（合計、調査根拠ページ記載の監査範囲）。

各画面について:
1. 人間: その画面を開き、Claude が提示する `WAVE3-CHECK` 項目に対して「一致 / 差異（具体）/ 新規項目あり」を報告。確認するのは **画面に実際に出ているか・日本語公式 UI でのラベル実文言・トグルや選択肢の有無・既定表示**（キー名/既定値/enum は Wave 2 がソース確定済みなので人間は見なくてよい）。
2. `hermes-docs-writer`: 差異があれば本文を修正。Wave 2 が「Wave 3 送り」にした警告ブロック判定も、実機で最終確証して削除/更新。
3. `hermes-accuracy-checker`: その画面に対応するページの frontmatter を `status: verified` + `verified: <当日 YYYY-MM-DD>` に昇格（`date +%F` で当日取得）。
4. 消化した `WAVE3-CHECK` 項目はチェック済みにする（ブロックは Wave 5 で除去するためマーカーは残してよいが、未確認項目が残る場合は明記）。

**昇格の鉄則**: 実機で確証が取れた画面・ページだけ `verified`。確認しきれない項目が残るページは `needs-review` 据え置き（橙バッジで公開されるが「要再確認」明示なので安全）。**verified へ一括昇格しない。**

ガイド3ファイル（`04_ガイド/*`）とリファレンスのうち実機照合で確定できるものも、根拠が揃った時点で昇格してよい（索引・調査根拠は Wave 5 で最終確定するので needs-review 据え置きが基本）。

## 完了条件（DoD・機械判定）

```bash
# 検証ゲート
npm run preprocess     # => ContentValidationError なし
# 実機確認できた画面が verified 昇格している（少なくとも1つ以上、理想は14画面）
grep -rl "status: verified" vault/03_設定/ | wc -l
# verified 昇格ページの verified 日付が当日に更新されている（目視 or grep）
grep -rn "verified: '$(date +%F)'" vault/03_設定/ | wc -l
```
加えて（人手確認）: 昇格したページはすべて実機照合済み。未確認項目が残るページは needs-review のまま。新画面発見は Wave 4 申し送りに記録。

## 失敗時の撤退

- 実機照合の途中で中断する場合、確認済み画面だけ verified 昇格して残りは needs-review 据え置き（部分完了が許容される設計）。マニフェストの行は `[/]` のままにし、実行ログに「どの画面まで確認済みか」を記録。
- 誤って verified 昇格したページを戻す: `git restore vault/03_設定/<該当>.md` の frontmatter 部分、または手で `status: needs-review` に戻す。
