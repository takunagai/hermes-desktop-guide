- **破壊的変更の過剰改変リスク**: `write_mode` / `write_approval` / `send_message` は vault に名前として存在しない（マニフェストのガード参照）。これらが v0.17 の **Desktop 設定 UI に露出している確証がソースで取れない**場合、既存記述を書き換えず、Stop And Ask か「変更履歴ページ送り（Wave 4）」として実行ログに記録する。**既存の承認モード `approvals.mode`（manual/smart/off）を書き換えない。**

## 進め方（共通）

各クラスタを **`hermes-accuracy-checker`（ソースでキー名・既定値・enum・ラベル・SHA を確定）→ `hermes-docs-writer`（本文を `needs-review` で起草）** の 2 段で処理する。クラスタ間は独立。各クラスタ完了後に `content-validator` で `npm run preprocess` + `npm test`。

## 失敗時の撤退

- preprocess/test が落ちたら、直前に編集したページの WikiLink（パス付き形式 `[[03_設定/12_MCP|MCP]]`）・コールアウト記法・frontmatter を確認。
- 反映を巻き戻す: `git restore vault/03_設定/<該当>.md`。
- ソースに辿り着けない（SHA のファイルが見つからない等）: リリースノート本文の範囲で確定できる事実だけ反映し、残りは実行ログに「Wave 3 実機確認待ち」と明記して継続（止めずに安全側で残す）。
