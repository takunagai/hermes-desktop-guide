# Hermes Desktop ガイド — IA を「設定特化」から「製品全般ハブ」へ再編

> **ステータス**: 計画（未実装）／**作成日**: 2026-06-08
> 設定特化のサイトを、Hermes Desktop 全般のドキュメントハブへ拡張するための IA 再編計画。

## Context（なぜこの変更をするか）

現状サイトは Hermes Desktop の**設定**に特化している（設定 14 ページで設定画面を 100% 網羅・約 97K 字、ガイド 3・リファレンス 3）。完成度は高いが、サイト名・リポジトリ名が「Hermes Desktop」全般を名乗るのに対し、コンテンツは設定限定で「Hermes Desktop とは / 導入 / 使い方」という上流の入口を欠いている。

ユーザーは Obsidian Vault を抄本としてサイト化する運用を**全体的に**広げたい意向。そこで IA の「器」を製品全般のハブへ広げ、設定は看板コンテンツとしてその一部に位置づける。中身（概要・導入・スキル・Obsidian 連携等）は段階的に追加する方針。

未デプロイ（Cloudflare 未設定）の今なら、フォルダ再採番に伴う URL 変更も被リンク切れの実害がゼロ。再編の最適タイミング。

## 決定事項（ユーザー確認済み）

1. **方向性**: 器を全般化・段階公開（今回は IA 骨格づくりが中心、本文は順次）
2. **設定の階層**: 設定をトップレベル化（現状のフラット 2 階層を維持。3 階層ネストはしない）
3. **URL/番号**: 今、再採番して整える（公開前の無傷な今に実施）

## 技術的根拠（なぜコード改修なしで動くか）

- **WikiLink**: preprocess は `urlMap` を `relNoExt` と `basename` の両方で構築し、解決時に「フルパス → なければ basename」でフォールバックする（`preprocess.ts:160-164`）。フォルダをリネームしても basename がユニークなら URL は新パスで正しく解決される。**ただし Obsidian 側はフルパス WikiLink だとフォルダ移動で未解決リンクになる**ため、A-2 のリンク追従が必要。
- **サイドバー順序**: グループは `group.key`（フォルダ名）を `localeCompare` でソート（`Layout.astro:73`）。`01_概要 < 02_導入 < 03_設定 < 04_ガイド < 05_リファレンス` で意図通り。見出しは数字プレフィックス除去（同 `:34`）で「概要 / 導入 / 設定 …」と表示。グループ内は `order` → `slug`（同 `:76-81`）。
- **トップレベルセクション追加**: フォルダを足すだけで自動グループ化。改修不要。

## 注意事項

- **Vault は別リポジトリ／別ディレクトリ**（`/Users/takna/Documents/vault-hermes`）。このサイトの git には絶対コミットしない。Vault 側の git 有無は実装時に確認し、コミットする場合は `cd` で cwd を切り替える（P0 ルール）。
- `src/content/docs/` は preprocess の再生成物。直接編集しない。
- 段階公開: 中身（`.md`）の無いフォルダは Astro が拾わずサイドバーにも出ない。空セクションは自然に非公開になる。

## 検証

1. `npx tsx scripts/preprocess.ts` → **`[Warning]` WikiLink 未解決が 0 件**であること
2. `npm run dev` → サイドバーが「概要 / 導入 / 設定 / ガイド / リファレンス」の順、見出しは数字なし表示
3. 設定ページ・ガイドの相互リンクをクリックし 404 が無いこと、URL が `/03_設定/...` 等になっていること
4. ホームの全リンクが有効、ヘッダー検索に新セクションが出ること
5. ダーク／ライト・モバイルドロワーで表示崩れが無いこと
6. `npm run build` 成功
7. （Obsidian）Vault を開き、未解決リンク（リンク切れ）が無いこと
