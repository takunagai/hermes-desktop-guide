## 目的

設定画面の詳細資料を中核にしながら、初めて利用する人が「製品を知る」「導入する」「最初の操作を行う」「問題を解決する」まで迷わず進めるHermes Desktop全般の日本語マニュアルサイトへ拡張する。

次の品質を満たす情報基盤を構築する。

- 閲覧者の目的に沿った情報設計
- フォルダ移動や並び替えで変わらない恒久URL
- 本文まで対象にした全文検索
- モバイル、キーボード、支援技術で利用できるナビゲーション
- 公開前にリンク切れやメタデータ不備を検出する自動検証
- Obsidian Vaultを正本として維持できる編集フロー

## 決定事項

1. サイト名を「Hermes Desktop ガイド」に変更する。
2. ドキュメント基盤を独自UIからAstro Starlightへ移行する。
3. Obsidian Vaultをコンテンツの正本として維持する。
4. 設定はトップレベルの主要セクションとして維持する。
5. ファイル名の数字はVault内の整理にだけ使い、公開URLには含めない。
6. 公開URLは各ページのfrontmatter `slug`で明示する。
7. WikiLinkはパス付き形式を基本とし、basenameだけのリンクに依存しない。
8. 内容のない「準備中」ページは公開しない。
9. 将来セクションは、公開可能な本文ができるまで追加しない。

## URL設計

### 原則

- `slug`は必須とし、ASCII小文字の恒久パスを指定する。
- Vaultのフォルダ名、ファイル名、表示順を変更しても`slug`は変更しない。
- `slug`の変更が必要な場合は旧URLからリダイレクトする。
- URLとページタイトルを分離し、日本語タイトルを変更してもURLを維持する。

例:

```yaml
---
title: モデル
description: Hermes Desktopで利用するモデル、コンテキスト長、フォールバックを設定する方法。
slug: settings/models
sidebar:
  order: 1
---
```

```text
03_設定/01_モデル.md → /settings/models/
04_ガイド/01_推奨設定プリセット.md → /guides/recommended-presets/
```

## コンテンツモデル

Starlight標準フィールドに加え、次のfrontmatterをスキーマで検証する。

| フィールド | 用途 |
|---|---|
| `title` | ページタイトル |
| `description` | 検索結果・SEO向けのページ要約 |
| `slug` | 恒久URL |
| `sidebar.order` | セクション内の表示順 |
| `draft` | 本番公開から除外 |
| `tags` | 検索・分類用キーワード |
| `audience` | 想定読者 |
| `platforms` | 対象OS |
| `status` | `verified`または`needs-review` |
| `hermes_version` | 確認した製品バージョン |
| `hermes_commit` | 確認したコミット |
| `verified` | 実機・根拠の確認日 |

### 公開基準

- `title`、`description`、`slug`がある。
- 本文だけで閲覧者の目的を少なくとも1つ達成できる。
- 「準備中」の見出しだけで公開しない。
- 手順には前提、操作、期待結果、失敗時の確認先を含める。
- バージョン依存情報には対象バージョンまたは根拠リンクを付ける。
- 未確認内容は断定せず、`status: needs-review`として明示する。

## WikiLink方針

- 別セクションへのリンクはルート基準のパス付きWikiLinkを使う。
- basenameだけのリンクは、basenameが全Vaultで一意な場合のみ許可する。
- preprocessはbasename重複、未解決リンク、重複slugをエラーとして停止する。
- 相対パス、アンカー、表示ラベル、表内のエスケープされた区切り文字を処理する。
- コードフェンス内のWikiLinkとコールアウトは変換しない。

## Starlight移行

次をStarlight標準機能へ置き換える。

- Pagefindによる本文・見出し・タグの全文検索
- モバイル対応検索
- サイドバーと現在地表示
- 前後ナビゲーション
- デスクトップ・モバイルの目次
- ダークモード
- Skip Link、キーボード操作、ARIAを含むアクセシブルなUI
- 404ページ
- canonical、OGP、sitemapの基盤
- `draft`ページの本番除外

カスタマイズはサイト名、説明、favicon、配色、ページ確認情報の表示に限定する。ObsidianコールアウトはpreprocessでStarlight Aside記法へ変換する。

## preprocessの責務

1. VaultのMarkdownを再帰収集する。
2. frontmatterと`slug`を検証する。
3. basenameと`slug`の重複を検出する。
4. WikiLinkを恒久URLのMarkdownリンクへ変換する。
5. ObsidianコールアウトをStarlight Asideへ変換する。
6. コードフェンス内は変換しない。
7. 検証成功後に`src/content/docs/`を再生成する。
8. コンテンツ整合性エラーでは終了コード1で停止する。

## アクセシビリティ要件

- WCAG 2.1 AAを目標とする。
- キーボードだけで検索、サイドバー、テーマ切り替え、本文リンクを操作できる。
- 見出し階層を論理的にする。
- 色だけで状態を伝えない。
- 通常テキストは4.5:1以上のコントラストを確保する。
- `prefers-reduced-motion`を尊重する。

## 検証と品質ゲート

実装完了条件は次のすべてが成功すること。

1. `npm run preprocess`
2. `npm run validate:content`
3. `npm run lint`
4. `npm run check`
5. `npm test`
6. `npm run build`
7. `npm run check:links`
8. Pagefindインデックス生成確認
9. デスクトップとモバイルの主要導線をブラウザ確認
10. キーボード操作とアクセシビリティ検査
11. `npm audit`結果の確認

## ロールバック

- Vaultは変更前にディレクトリ全体を日時付きでコピーする。
- サイトコードはGitの変更差分から戻せる状態を維持する。
- IA移行に失敗した場合はVaultバックアップを正本へ戻す。
- `src/content/docs/`は生成物なのでロールバック対象にしない。
